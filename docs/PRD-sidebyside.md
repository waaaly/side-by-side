# PRD: SideBySide (SBS) — 情侣专属共生数字空间

## Overview

### Problem Statement

情侣共同生活中存在三个核心痛点，现有工具（独立记账 App、菜谱 App、日历 App）彼此孤立，无法在一个界面内形成联动闭环：

1. **💸 花钱如流水，月底不知道花在哪**：两人日常开销场景多（买菜、外卖、约会、房租、猫狗），每笔记了但月底只是一串数字，没有"超支提醒→分类占比→优化建议"的完整链路。缺乏数据驱动的消费洞察，无法真正达到开源节流的目的。
2. **🍽️ 每天不知道吃什么，也没记录下一起吃过的美好**：选择困难是日常高频痛点；同时，一起做过的美食、去过的好店，除了手机相册里零散的照片外没有系统化的"美食记忆档案"。
3. **📅 重要日子全靠脑子记，漏了就是矛盾**：生理期、双方父母生日、纪念日、约会日程——这些日期分散在各自手机里，没有双人同步的日历中枢，也没有主动提醒和行动建议。

Three problems, one app.

### Solution Summary

SideBySide 是一款基于 Next.js + Firebase 的 PWA 应用，通过底部五 Tab 导航整合四大页面模块 + 一个 AI Agent 层：

| 模块 | 路由 | 定位 | 解决什么问题 |
|------|------|------|-------------|
| **📋 账单** | `/` | 收支记录 + 月度消费洞察 | 知道钱花在哪、花得是否合理、下月怎么优化 |
| **🍳 食谱** | `/kitchen` | 吃什么 + 美食记忆 | 盲盒选菜 + 拍照记录一起吃过的好味道 |
| **➕ 快速记账** | 中心悬浮按钮（非页面） | 一键快速记账入口 | 随时记一笔，无需切换到账单页 |
| **📅 日历** | `/calendar` | 双人日程中枢 | 生理期、生日、纪念日、日程，一处同步 |
| **⚙️ 设置** | `/settings` | 系统管理 | 分类管理、食谱管理、预算设置、个性化配置 |

**🤖 AI Agent** 作为贯穿层，为所有模块提供智能能力。

五个部分互相联动，形成完整闭环：「日历触发经期/生日→ AI 推荐食谱→ 一键加入买菜清单→ 买菜转记账→ 月底 AI 分析给出省钱建议」。

### Target Users

| 角色 | 画像 | 核心诉求 |
|------|------|---------|
| 女生 | 主动记账、主导食谱库建设 | 财务透明、经期有人关心、重要日子不被忘记 |
| 男生 | 需要被引导/提醒的角色 | 知道今天吃什么、知道该做什么、不被埋怨 |
| 两人共同 | 一起生活的情侣 | 减少因信息不对称产生的不必要摩擦 |

## Goals & Success Metrics

### Goals

1. **财务透明化**——每笔支出都有分类归属，月底能清晰看到"钱去哪了"，并能收到 AI 给出的下月优化建议
2. **选择消失术**——"今天吃什么"从每日难题变成 3 秒出答案的乐趣，同时积累一本两人专属的"美食回忆录"
3. **零遗忘日历**——所有重要日期（生理期、生日、纪念日、约会）双人实时同步，到点主动提醒

### Success Metrics

| Metric | Current Baseline | Target | Timeline |
|--------|-----------------|--------|----------|
| 月度消费分类覆盖率 (被打上分类的支出占比) | — | >95% | 上线后 30 天 |
| 用户每月主动查看统计页面的次数 | 0 | ≥4 次/月 | 上线后 60 天 |
| 盲盒使用频次 | — | ≥每天 1 次/对 | 上线后 30 天 |
| 每月新增美食照片数 | — | ≥10 张/对 | 上线后 60 天 |
| 日历事件遗漏率 | 100%（无系统） | <5% | 上线后 30 天 |
| AI 财务分析阅读率 | — | >70%（看过并采取了行动） | 上线后 90 天 |
| 月度留存率 | 0% | >70% | 上线后 90 天 |

### Non-Goals

- 不做 AA 分摊计算（不关心谁付了多少，只关心总共花了多少、花在哪）
- 不做通用社交功能（无好友系统、无公开动态）
- 不做线下配送、电商或外卖下单
- 不替换完整经期健康 App（仅做标记与提醒，不做周期预测算法）
- 不做通用日历替换（仅管理双人相关事件，不导入 iCloud/Google Calendar）
- 不做多人账本（仅支持双人）

## Scope

### In Scope (P0 — MVP)

**记账模块**

**分类体系（32 分类，4 组 8 列）：**

| 组 | 分类 | 分类 | 分类 | 分类 | 分类 | 分类 | 分类 | 分类 |
|----|------|------|------|------|------|------|------|------|
| 🍔 饮食 | 餐饮 | 蔬菜 | 水果 | 饮料 | 烟酒 | 零食 | — | — |
| 🏠 居住 | 住房 | 居家 | 日用 | 通讯 | 水电（预留） | — | — | — |
| 🚗 出行 | 交通 | 汽车 | 旅行 | 酒店 | — | — | — | — |
| 🛍️ 消费 | 购物 | 服饰 | 美容 | 数码 | 电子 | 娱乐 | 运动 | 社交 |
| 📚 成长 | 书籍 | 学习 | — | — | — | — | — | — |
| ❤️ 人情 | 长辈 | 孩子 | 礼金 | 宠物 | — | — | — | — |
| 🧰 杂项 | 医疗 | 快递 | 网络 | 理财 | — | — | — | — |

- 快速记账：从底部弹出的抽屉式选择器中选择分类（32 种）、输入金额（数字键盘）、填写备注
- 流水编辑：点击已有流水条目进入编辑模式，可修改金额、分类、备注、**日期**（默认展示录入时的日期，用户可调，用于补录忘记的消费）
- 首页预算卡片：预设月预算总额、已花金额、环形进度条（绿 ≤80% 橙 80%-100% 红 >100%）
- 最近流水列表：分类图标、描述、金额、日期，支持点击编辑
- 多维度统计页面：可按**人员**（女方/男方/双方）、**时间粒度**（年/季/月/周/日）、**消费排行**（分类金额降序）切换查看

**食谱模块**
- 食谱库 CRUD：菜名、emoji、食材列表、标签（如"快手菜""暖宫""30分钟"）
- "今天吃什么"盲盒：从食谱库随机抽取，spring 动画展示，可重新抽取
- 买菜实时清单：双人同步添加/勾选/删除，勾选后有划线动画
- 美食记忆墙（P0 MVP 文字版）：时间线展示"某年某月某日·我们吃了 XXX"，带 emoji

**日历模块**
- 月视图日历
- 生理期标记：点击日期标记"开始"/"结束"，粉色高亮区间
- 纪念日标记：自定义日期 + 标题（如"在一起 1000 天"），爱心高亮
- 生日标记：自定义日期 + 姓名，蛋糕图标高亮
- 所有事件双人实时同步可见

**设置模块**
- 分类管理：查看/编辑 32 个消费分类的 emoji 图标、名称、启用/停用，可自定义排序
- 食谱管理：查看/编辑/删除食谱库中的所有菜谱，管理标签体系
- 预算设置：设定月度总预算上限，可为单个分类设定独立预算上限
- 日历事件管理：查看/编辑/删除所有日历事件（生理期、纪念日、生日、日程）
- 月度预算卡片文案定制：可修改预算进度条上的显示文案

**AI Agent**
- 基础版：接入 LLM API，接收结构化数据后生成自然语言文本
- 月度财务分析：汇总本月分类支出，生成可读报告 + 3 条优化建议
- 食谱推荐：根据标签（经期、时令、快手）从已有食谱库推荐
- 日程提醒：日历事件在 T-1 天时，首页展示提醒卡片

**全局**
- 底部导航栏：5 个 Tab（📋 账单 / 🍳 食谱 / ➕ 快速记账 / 📅 日历 / ⚙️ 设置），➕ 居中放大突出显示，点击直接弹出记账分类选择器，其余 4 个 Tab 切换对应页面
- 导航栏样式：➕ 使用凸起浮动按钮（带阴影 + 背景色渐变），其他四 Tab 为标准扁平图标 + 文字
- iOS PWA 全屏适配：safe-area-inset、状态栏颜色、禁止全局滚动
- 所有点击按钮 `active:scale-95 transition` 物理触感
- 页面 `h-screen` + `overflow-y-auto` 限制，无全局滚动

### In Scope (P1 — v1.1)

- AI 财务分析升级：对比上月环比、检出异常支出、给出具体可执行的省钱方案
- AI 食谱推荐升级：根据季节食材、冰箱库存（手填）推荐
- 美食拍照上传：相机/相册选择，图片附在美食记忆墙中
- 日程事件管理：自定义约会/日程，双人确认状态
- 预算分类设置：每个分类可独立设定预算上限
- 买菜清单一键转记账：勾选的清单项直接生成记账条目
- 经济模式过渡动画

### Out of Scope

- 第三方登录/注册系统（初始版本使用本地存储 + 手动邀请码双人绑定）
- 推送通知（iOS PWA 无原生推送，v1.1 考虑 Service Worker + Web Push）
- 多语言支持（仅中文）
- AA 分摊计算功能
- 照片云存储/CDN（v1.1 考虑 Firebase Storage）
- 与外部日历同步（Google/Apple Calendar）
- AI Agent 的自定义训练/知识库植入

### Future Considerations (P2+)

| Feature | Description | Defer Rationale |

| 触发时机 | 出现位置 | 卡片内容 |
|---------|---------|---------|
| 每月 1 号首次打开 App | 记账页顶部 | "🤖 上月消费分析：餐饮占比最高(35%)，建议本月减少外出就餐，预计可省 ¥800" |
| 经期 Day 1 | 厨房页盲盒上方 | "🤖 今天推荐暖宫食谱：红糖姜茶、红枣炖蛋，已帮你优先筛选" |
| 纪念日前 1 天 | 日历页 + 首页顶部 | "🤖 明天是纪念日💕 预算还剩 ¥300，推荐：在家做一顿烛光晚餐" |
| 生日前 2 天 | 日历页 | "🤖 妈妈的生日在后天，建议提前订蛋糕🎂" |
| 预算超支时 | 首页预算卡片下方 | "🤖 本月已超支 ¥200，超支主要来自「外出约会」，建议月底前减少外食" |

AI 建议均**可关闭**（右上角 X），用户可反馈 👍/👎。所有 AI 内容使用流式打字效果输出，减少等待焦虑。

### Edge Cases & Error States

| Scenario | Expected Behavior |
|----------|------------------|
| **两人同时录入一笔账** | 各存各的，列表按时间戳合并排序，不冲突 |
| **盲盒食谱库为空** | 显示空状态插画 + 引导文案："还没有菜谱哦，先去添加一道吧 →" |
| **买菜清单离线操作** | 本地缓存优先，联网后自动合并同步，冲突时保留最新 |
| **预算设为 0 或负数** | 前端校验不允许 ≤0，默认 5000 |
| **生理期跨月标记** | 开始日期在月末，结束日期在下月，日历视图自动衔接 |
| **日历事件删除** | 单次事件直接删除；重复事件（生日/纪念日）弹窗确认"仅删除今年"或"删除所有" |
| **AI API 调用失败/超时** | 回退显示"AI 分析暂不可用"，使用本地缓存的上次报告，不阻塞页面 |
| **AI 生成内容不准确** | 用户可反馈 👍/👎，连续 👎 3 次该类型建议暂时降权 |
| **美食照片上传失败** | 图片暂存本地，标记"上传中"，下次联网自动重试 |
| **双人同时编辑同一份清单** | Firestore 实时同步，最后写入胜出，短暂显示"清单已更新"提示 |
| **纪念日每年重复** | 系统自动在次年生成新事件，支持提前编辑/删除特定年份 |
| **预算分类修改后历史数据** | 分类变更不影响历史数据，统计时仍按录入时分类计算 |
| **对方删除日历事件** | 本端收到事件删除通知，事件从日历移除，弹窗提示"对方取消了 X 事件" |

### Constraints

- **PWA 推送限制**：iOS PWA 不支持原生推送，日程提醒使用首页顶部提醒卡片（被动式），打开 App 即见
- **AI API 成本**：每次分析约 500-1000 tokens，按每月 1 次财务分析 + 按需推荐估算，月度成本可控 (<$1/对/月)
- **离线可用性**：核心操作（记账录入、清单勾选、日历查看）需支持离线，使用 Firestore 离线持久化 + IndexedDB 缓存
- **首屏加载**：PWA 必须控制在 1MB 以内，使用 Next.js 动态导入 + React.lazy 拆分各模块
- **模型选择**：AI 推荐使用 `gpt-4o-mini` 或 `claude-3-haiku` 级别模型，平衡质量与成本

### AI Prompt 设计（概念）

**Prompt 1：AI 智能归类 —— 将 32 种小分类归并为大分类**

```
你是一个智能归类引擎。
以下是一对情侣本月消费中出现的所有小分类列表，请将它们归并为若干个大分类（3-8 个）。
归并原则：语义相近的归为一类（如"餐饮""饮料""零食"→"饮食"），
大分类名称需简洁易懂（2 字），并为每个大分类选择一个 emoji。

小分类列表：
{{出现的分类名数组}}

输出格式（JSON）：
[
  { "groupName": "饮食", "emoji": "🍔", "includes": ["餐饮", "饮料", "零食", "蔬菜", "水果"] },
  ...
]
```

**Prompt 2：月度财务分析（基于 AI 归并后的大分类 + 原始小分类双层分析）**

```
你是一个贴心的财务助手，正在为一对情侣分析本月消费。

【原始 32 分类数据（已按 AI 归类分组）】：
{{归类后的分类数据 JSON，包含 groupName 和其下的小分类汇总}}

【上月对比数据】：
{{上月大分类汇总 JSON}}

月度总预算：{{总预算}}
已花费：{{已花费}}
剩余：{{剩余}}

请输出：
1. 【一句话总结】本月的消费概况（语气温暖、鼓励）
2. 【大分类层面分析】占比最高的前 3 个大分类，推荐占比范围
3. 【小分类层面分析】大分类内部，哪些小分类花得最多
4. 【环比变化】相比上月，显著增加或减少的分类（标出变化幅度）
5. 【异常检测】单笔超过该分类平均 3 倍的支出，标红提醒
6. 【3 条优化建议】具体可操作的下月省钱建议

格式：用 markdown 但仅使用纯文本，每条建议不超过 30 字。
语气：温暖、鼓励，像朋友在聊天，不要说教。
```

**Prompt 3：财务统计问答（用户交互式查询）**

```
用户问了一个关于消费统计的问题。根据以下数据回答：
{{当前用户筛选状态：人员/时间粒度/排序方式}}
{{对应的消费数据 JSON}}

回答要求：
1. 直接回答用户问题，给出具体数字
2. 如果用户没有具体问，按当前筛选维度给出 Top 5 排行
3. 语气轻松自然
```

### Design Principles

1. **温暖优先**：所有文案使用"我们"而非"你"，鼓励而非说教
2. **数据驱动**：每一笔记录最终都能在月底转换为有价值的消费洞察
3. **选择最小化**：能预设的不让用户选（分类预设、盲盒随机），降低每日使用摩擦
4. **AI 可见但克制**：AI 建议永远以卡片形式出现，不打断主流程，可关闭可反馈
5. **离线第一**：核心操作不依赖网络，同步是锦上添花而非必须
